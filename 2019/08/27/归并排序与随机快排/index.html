关于mergeSort的递归方式与非递归方式的实现，以及从Partition来聊一聊快速排序的升级过程。

归并排序与随机快排

递归方式实现mergeSort

  1. 整体是递归，左边排好序+右边排好序+merge让整体有序

话不多说，直接上代码：

//递归代码
public static void process(int[] arr, int L, int R) {
    if(L == R) {
        return;
    }
    int mid = L +((R - L) >> 1);
    process(arr, L, mid);
    process(arr, mid + 1, R);
    merge(arr, L, mid, R);
}
//这段递归代码很好理解，上一篇文章中解析过一个递归行为，和这个相似。


public static void merge(int[] arr, int L, int M, int R) {
    int[] help = new int[R - L + 1]; //这个help数组的长度就是L到R之间的元素个数,用来存放合并之后的数组元素
    int i = 0;                      //i是拿给help数组使用的
    int p1 = L;                    //左边使用的指针
    int p2 = M + 1;                //右边使用的指针
    while(p1 < M && p2 < R) {
        help[i++] = arr[p1] < arr[p2] ? arr[p1++] : arr[p2++]; //这一行代码搞定了数组合并这个看起来貌似很复杂的问题                                                                                             
    }
    //要么p1越界了，要么p2越界了，上面的while循环就会break
    while(p1 <= M) {
        help[i++] = arr[p1++];
    }
    while(p2 <= R) {
        help[i++] = arr[p2++];
    }
    for(i = 0; i < help.length; i++) {
        arr[L + i] = help[i];
    }
    //非递归方法实现
    
    
}

非递归方式实现mergeSort：

public static void mergeSort2(int[] arr) {
    if(arr == null || arr.length < 2) {
        return;
    }
    int N = arr. length;
    int mergeSize = 1;  //一组是两倍的mergeSize，分别是左边的mergeSize和右边的mergeSize
    				  //可以解释为当前有序的左组长度
    
    while(mergeSize < N) {
        int L = 0;
        while(L < N) {
            //现在我令一个M，从L...M这段范围是左组大小(左组长度当然是mergeSize)，所以我可以把M算出来
            int M =  L + mergeSize - 1;
            if(M > N) {
                //如果我发现左组的范围已经大于了L到R的整体范围，break
                break;
            }
            //L...M, M+1...R. 下面这行代码是判定右边的merge区域的右边界在哪里
            int R = Math.min(M + mergeSize, N -1);
            merge(arr, L, M, R);
            L = R + 1;
        }
        //防止溢出的安全语句
        if(mergeSize > N/2) {
            break;
        }
        mergeSize <<= 1; //左移一位，相当于乘以二
    }
}

  1. 让其整体有序的过程里用了排外序方法

  2. 利用master公式来求解时间复杂度

    时间复杂度：T = T(N) = aT(N/b) + O(N^d)；

    归并排序的时间复杂度：T = 2(N/2) +O(N)

    满足logab = d = 1，所以其时间复杂度为N^d * logN = nlogn

归并排序的时间复杂度小于前三个排序的N^2，因为前三个排序（比如冒泡排序，存在浪费比较行为的问题）

用常见面试题来深入理解一下归并排序的精髓

在一个数组中，一个数左边比它小的数的总和，叫数的小和，所有数的小和累加起来，叫数组小和，求数组小和

这道题怎么做呢？用归并排序

跟归并排序的流程一模一样，其它地方不发生变动，唯一发生变动的过程就是merge的过程。

说一下merge过程的规则

首先我们知道归并排序的merge过程是准备了一个help数组的，然后呢被排序的数组中还分了左组和右组，左组和右组中各有一个指针，当左右指针分别指向左组和右组中第一个元素时，开始比较二者的大小，小的会被优先放入help数组，同时被取出元素的的那个数组的指针++，help数组的指针++。

merge过程中的新增规则：如果放入元素比较时左组的元素值小于右组的元素值，（因为此时左右组都是已经升序排好序了的），那么左组的这个元素的值，一定比右组指针此时指向的元素 /之后的元素的值 / 都要小，那么我们可以直接通过数组下标计算出右组中 /大于此时左组指针指向的左组元素 / 的元素个数，然后将左组指针此时所指元素放入help数组。如果左组指针和右组指针所指元素的值相等，优先放入右组元素，不记录。如果左组指针所指元素比右组指针所指元素的值大，放入右组元素，不记录。

求小和题目的代码：

//L...R既要排好序，也要求小和返回
//要求所有merge时，产生的小和累加
//左侧排序的时候会产生很多merge行为
//右侧排序的时候也会产生许多merge行为
//最后左右总merge时，还会产生merge行为
//总的来说就是我只管我在merge的时候产生多少	
public static int process(int[] arr, int l, int r) {
    if(l == r) {
        return 0;
}
    int mid = 1 + ((r -1) >> 1); 
    return
        	process(arr, 1, mid)
        	+
        	process(arr, mid + 1, r);
    		+
            merge(arr, 1, mid, r);
        	
    
//这个是merge方法，和归并的几乎一模一样，那个res三目运算符一句直接完成小和相加的任务，无敌。
public static int merge(int[] arr, int L, int m, int r) {
    int[] help = new int [r - L + 1];
    int i = 0;
    int p1 = L;
    int p2 = m + 1;
    int res = 0;
    while(p1 <= m && p2 <= r) {
        res += arr[p1] <arr[p2] ? [r - p2 + 1] * arr[p1] : 0;
        help[i++] = arr[p1] < arr[p2] ? arr[p1++] : arr[p2++];
    }
    while(p1 <= m) {
        help[i++] = arr[p1++];
    }
    while(p2 <= r) {
        help[i++] = arr[p2++];
    }
    for(int i = 0; i < help.length; i++) {
        arr[L + i] = help[i];
    }
    return res;
}

快速排序

Partition过程（分区过程）

给定一个数组arr，和一个整数num。请把小于等于num的数放在数组的左边，大于num的数放在数组的右边。（不要求左右两边有序，但是要分成左右两边）

要求额外空间复杂度O(1)，时间复杂度O(N)

给你一个数组，让你把这个数组中小于4的数放在左边，大于4的数放在右边，等于4的数放在中间

荷兰国旗问题

//在arr[L...R]上玩荷兰国旗问题，一律以arr[R]做划分值	
//划分成左边是小于arr[R]的，等于的在中间，大于的在右边
//返回等于区域的左边和右边位置即可
public static int[] netherlandsFlag(int[] arr, int L, int R) {
    if(L > R) {
        return new int[] {(-1, -1)};
    }   //L > R，没区域可分，直接返回(-1,-1)
    	if(L == R) {
            return new int[] {L, R};
        }	//等于区域只有一个数，L==R
    //上面的是basecase，下面的就是荷兰国旗问题的代码，就这么短
    int less = L - 1;  //小于区域的右边界
    int more = R;      //大于区域左边界
    index = L;         //index是遍历的指针，从L开始遍历，一直到index撞上大于区域边界之后，遍历结束
    while(index < more) {
        if(arr[index] == arr[R]) {
            index++;  //如果等于，什么也不干，index++
        } else if(arr[index] < arr[R]) {
            swap(arr, index++, ++less);
            //如果当前位置的数小于指定的数，做三个操作：
            //1.把当前的数和小于区域的右一个数交换  2.当前位置++  3.小于区域++
            //先把less代表小于区域，先把index和小于区域的右一个数（++less）交换，然后当前位置++（index++）
            //然后因为之前做了++less，所以小于区域已经++了
            //三个说法一行代码，左神tql
    	} else { //最后一个分支一定是中了大于的
            	 //把i位置的数和大于区域的左一个做交换，然后大于区域左移一个位置，i停在原地
            swap(arr, index, --more);
        }
    }  
    swap(arr, more, R);//最后要把比较的标杆数放中间，也就是相等区域的最后一个，大于区域的第一个，这样刚好
    				 //所以此时交换标杆数和大于区域的第一个数
    				 //换完之后它就整体都符合要求了
    return new int[] {less + 1, more};
    //返回等于区域的左右边界，因为最后等于区域多了一个数在more位置，所以more不用-1
}

讲这个干啥？引出我们的快速排序

快速排序1.0版本

标杆数X最终放在中间就不用再参与排序了，左右分别递归。每一次排序搞定一个数，总有搞定完的时候。这就是快排1.0版本，是最常见的版本，也是我们经常在考研数据结构复习指导书和严书上看到的版本。（当然也是最挫的版本）

public static void quickSort1(int[] arr) {
    if(arr == null || arr.length < 2) {
        return;
    }
    process1(arr, 0, arr.length - 1);
}
public static void process1(int[] arr, int L, int R) {
    if(L >= R) {
        return;
    }
    //在L...R上玩partition，用arr[R]做划分 [<arr[r]  ==arr[r]  >arr[r] ]
    //partition方法返回值的意思是arr[R]的最终在的位置
    int M = partition(arr, L, R);
    //每次会搞定一个数，M位置的数，以后就不会再碰了，分别左右递归。
    //总有排序完的时候
    process1(arr, L, M - 1);
    process1(arr, M + 1, R);
}

快速排序2.0版本

快排2.0版本采用上面提到的荷兰国旗划分法，与1.0版本的区别就在于1.0版本每次只能搞定一个数，而2.0版本每次能搞定与x相等的一批数（在大于和小于区域递归，等于区域不用再参与排序）。直接上代码：

public static void quickSort1(int[] arr) {
    if(arr == null || arr.length < 2) {
        return;
    }
    process2(arr, 0, arr.length - 1);
}

public static void process2(int[] arr, int L, int R) {
    if(arr == null || arr.length < 2) {
        return;
    }
    int[] equalArea = netherlandFlag(arr, L, R);
    process2(arr, L, equalArea[0] - 1);
    process2(arr, equalArea[1] + 1, R);
    //荷兰国旗函数返回的是等于区域的左右边界的数组，数组里有两个值，第一个是左边界，第二个是右边界
}

如果数组本来就是升序有序的（最坏情况），快排1.0和快排2.0版本每次都只能解决一个数，最后的时间复杂度是O(N^2)，都挺挫的。

接下来，3.0版本，特简单

快速排序3.0版本

在数组中随机选一个数i，与最后一个位置的数arr[R]交换，然后用i做划分，其它所有过程与快排2.0一样。这就是最经典的快排。

public static void quickSort3(int[] arr) {
    if(arr == null || arr.length < 2) {
        return;
    }
    process3(arr, 0, arr.length - 1);
}

public static void process3(int[] arr, int L, int R) {
    if(arr == null || arr.length < 2) {
        return;
    }
    //第一步，随机选一个位置，在L...R上随机选一个位置，然后人为的让它与R位置做交换
    //然后剩下的步骤与2.0版本一模一样
    swap(arr, L + (int)(Math.random() * (R - L + 1)), R);
    int[] equalArea = netherlandFlag(arr, L, R);
    process3(arr, L, equalArea[0] - 1);
    process3(arr, equalArea[1] + 1, R);
    //荷兰国旗函数返回的是等于区域的左右边界的数组，数组里有两个值，第一个是左边界，第二个是右边界
}

为什么这么一干，它就变成O(N^logN)了？前面已经说过了坏情况（整体有序且不重复且标杆数在最后），那什么是好情况呢？你选的标杆num左右两侧规模差不多的时候（都为N/2）

T(N) = 2 * T(N/2) + O(N)→partition的操作（或netherlandsFlag操作），刚刚看完归并排序，这个式子的时间复杂度就是NlogN

差情况就是num打的很偏，让左右两侧规模悬殊比较大的情况，好情况就是num打的很正，左右两边规模差不多大，复杂度就好。所以，想想我们随机选一个数，当N很大时，不可能每次都命中最差情况，只能说命中最差情况是个概率事件

当num来到最左侧时候（右侧全部都要排，T(N)为N^2)，num也可能来到距离开头1/3的位置，此时T(N) = T(1/3 * N) + T(2/3 * N) + O(N)

num也可能来到距离开头的3/7处，T(N) = T(3/7 * N) + T(4/7 * N) + O(N)

也有可能来到1/5处，4/5处，1/6处……等等，每种情况都有可能考虑到，但是每种情况的权重只占1/n。那么数学家把所有情况做了一个概率累加，得到的期望是O(nlogn)

解决一个小疑惑：之前不是说估计时间复杂度要按最差情况来估计吗？为什么这里又取平均情况？注意：比如，在升序插入排序中，最差情况是654321，此时的差情况是必命中的！也就是说这种情况下你只能排出T(N) = O(N^2)的算法，但是，快排呢？在快排中，即使此时的数组情况是654321，但差情况也只是一个概率，也就是是说数组为654321的时候，num可能会打在中间，造成T = nlogn，快排并不是差情况必命中。

上面的这个式子：T(N) = T(1/3 * N) + T(2/3 * N) + O(N)，这是master公式没法解的，master公式必须要子问题的规模一样才行，这个式子算复杂度还挺麻烦的，它是利用假设一个复杂度，然后看结果是否收敛于这个复杂度的方式，在算法导论上有，这里就不展开了。

注意，快排的额外空间复杂度是O(logn)（好情况下）

快排也可以实现成迭代的，但是你每次返回的中间数组的边界 [L,R] 自己也要记下来，你让递归来帮你记要方便些

递归空间就是每一次你用来记划分值位置的空间，这个东西是你省不掉的。是logn，和二叉树的高度一样，好情况下，左侧记完之后可以把空间拿给右侧，所以此时高度是多少，空间复杂度就是多少。

那么差情况呢？那就是O(N)了

每个位置的情况都要记录。所以最终的时间复杂度又收敛到logn去了，和时间复杂度一样，每种情况的概率都是1/n
